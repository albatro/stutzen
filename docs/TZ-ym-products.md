# ТЗ: Витрина товаров Яндекс.Маркета с ценами, остатками и комиссиями

## 1. Цель
Сделать внутренний инструмент, который по API кабинета Яндекс.Маркета (далее — ЯМ) подтягивает все наши товары и выводит их на одной странице в виде таблицы. По каждому товару должны быть видны: текущая цена, остаток на складе и комиссия ЯМ.

Это **первый этап**. На втором этапе через этот же интерфейс мы будем массово отправлять обратно в ЯМ новые цены и остатки — расчёт новых цен будет опираться на полученные комиссии. Поэтому архитектура должна сразу закладывать возможность писать в ЯМ, а не только читать.

## 2. Что есть на входе
- Доступ к API кабинета продавца на Я.Маркете (OAuth-токен либо API-ключ — уточнить тип).
- `businessId` кабинета и `campaignId` магазина (нужны для большинства методов ЯМ API).
- Список SKU = то, что заведено в самом кабинете (отдельного источника правды у нас нет, ЯМ — мастер).

## 3. Источники данных в ЯМ API
Все методы — из официального API партнёра ЯМ (`https://api.partner.market.yandex.ru`).

| Что берём | Метод | Где использовать |
|---|---|---|
| Карточки товаров (название, SKU, картинка, категория) | `POST /businesses/{businessId}/offer-mappings` | базовый список товаров |
| Цены | `POST /businesses/{businessId}/offer-prices` или `POST /campaigns/{campaignId}/offer-prices` | колонка «Цена» |
| Остатки на складе | `POST /campaigns/{campaignId}/offers/stocks` | колонка «Остаток» |
| Комиссии / тарифы ЯМ | `POST /tariffs/calculate` (расчёт по конкретным товарам с учётом цены/категории) | колонка «Комиссия ЯМ» |

Все методы — постраничные (`page_token` / `limit` до 200). Нужна сквозная пагинация с защитой от зацикливания.

## 4. Функциональные требования

### 4.1. Синхронизация (бэкенд)
- Кнопка «Обновить» на странице запускает синхронизацию.
- Также — фоновая синхронизация по расписанию (cron, например, раз в час; интервал — в конфиге).
- Один запуск синхронизации:
  1. Тянет полный список офферов через `offer-mappings` с пагинацией.
  2. Параллельно (или последовательно — по тому, что разрешает rate-limit ЯМ) тянет цены, остатки, комиссии для тех же SKU.
  3. Складывает результат в локальную БД (см. п.7).
  4. Логирует: сколько товаров обработано, сколько ошибок, сколько времени заняло.
- Обработка ошибок ЯМ:
  - 429 (rate limit) — экспоненциальный бэк-офф + ретрай.
  - 5xx — ретрай до 3 раз.
  - 4xx (кроме 429) — записать в лог по конкретному SKU, не валить весь прогон.

### 4.2. Страница «Товары» (фронтенд)
Одна страница, таблица со следующими колонками:

| Колонка | Источник | Примечание |
|---|---|---|
| Фото | offer-mappings | миниатюра |
| Название | offer-mappings | + ссылка на карточку в ЯМ |
| SKU продавца (`shopSku`) | offer-mappings | |
| SKU маркета (`marketSku`) | offer-mappings | |
| Категория | offer-mappings | |
| Цена, ₽ | offer-prices | текущая базовая цена в кабинете |
| Остаток, шт | stocks | сумма по всем складам (плюс tooltip — разбивка по складам) |
| Комиссия ЯМ, ₽ | tariffs/calculate | абсолютная сумма комиссии при текущей цене |
| Комиссия ЯМ, % | вычисляемая | `commission / price * 100`, округление до 0.1 |
| Обновлено | из БД | дата последней успешной синхронизации этого товара |

Возможности таблицы:
- Поиск по названию / SKU.
- Фильтр по категории.
- Сортировка по любой колонке.
- Пагинация (или виртуальный скролл) — товаров может быть тысячи.
- Индикатор «идёт синхронизация» вверху страницы.

### 4.3. Задел под второй этап (не делаем сейчас, но не мешаем)
- Структура БД должна позволить хранить «целевую цену» и «целевой остаток» отдельно от текущих.
- Слой работы с ЯМ API должен иметь методы `updatePrices(...)` и `updateStocks(...)` — пусть пока пустые / заглушки, но публичный интерфейс уже описан.

## 5. Не входит в этап 1
- Любая запись в ЯМ (только чтение).
- Бизнес-логика расчёта новых цен.
- Авторизация пользователей (инструмент внутренний, за VPN/баз. авторизацией реверс-прокси).
- История изменений цен и остатков (нужна будет на этапе 2 — заложить в схему БД, но UI не делать).

## 6. Нефункциональные требования
- Rate limit ЯМ: уважать, не более N RPS на метод (взять из доки на момент разработки).
- Полная синхронизация ~10к товаров должна укладываться в 5 минут.
- Страница открывается за <1с (всё читается из локальной БД, не дёргая ЯМ в реальном времени).
- Логи синхронизации хранятся минимум 14 дней.

## 7. Архитектура (предлагаемая)
- **Бэкенд:** Node.js (TypeScript) + Express/Fastify. Альтернатива — Python + FastAPI. Уточнить.
- **БД:** PostgreSQL. Таблицы: `offers`, `prices`, `stocks`, `commissions`, `sync_runs`.
- **Очередь / шедулер:** встроенный cron-планировщик на старте (BullMQ / APScheduler — если потребуется параллелизация).
- **Фронт:** один SPA-экран на React (или серверный рендеринг через Next.js) — таблица + кнопка «Обновить».
- **Конфиг:** `.env` с `YM_OAUTH_TOKEN`, `YM_BUSINESS_ID`, `YM_CAMPAIGN_ID`, `DATABASE_URL`, `SYNC_CRON`.

## 8. Этапы работ
1. **Каркас проекта** — репозиторий, линтер, Docker-compose с Postgres.
2. **Клиент ЯМ API** — типизированная обёртка над четырьмя нужными методами + пагинация + ретраи.
3. **Схема БД и миграции.**
4. **Сервис синхронизации** — ручной запуск через CLI / endpoint.
5. **API для фронта** — `GET /api/offers` с поиском/фильтром/сортировкой/пагинацией.
6. **UI** — таблица + кнопка «Обновить» + индикатор синхронизации.
7. **Шедулер** — автозапуск синхронизации по cron.
8. **README** — как запустить, как получить токены ЯМ, что в `.env`.

## 9. Что нужно от заказчика, чтобы начать
1. Тип авторизации в кабинете ЯМ: OAuth или новый API-ключ продавца? Сам токен.
2. `businessId` и `campaignId`.
3. Подтверждение стека (Node/Python, Postgres) — или указать свой.
4. Где будет хоститься (свой сервер / облако)? Это влияет на деплой.
5. Сколько примерно товаров в кабинете — чтобы оценить нагрузку и пагинацию.
6. Нужно ли уже на этапе 1 различать цены по разным `campaignId` (если магазинов несколько), или кабинет один.

## 10. Критерий приёмки этапа 1
- На странице видны все товары из кабинета.
- По каждому — корректные цена, остаток, комиссия (сверяем выборочно с кабинетом ЯМ).
- Кнопка «Обновить» обновляет данные без перезагрузки страницы.
- Cron автоматически обновляет данные раз в час (или по согласованному интервалу).
- Документация в README позволяет другому разработчику развернуть проект с нуля.
